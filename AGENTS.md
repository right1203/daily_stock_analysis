# AGENTS.md

이 문서는 이 저장소에서 개발, 이슈 분석, PR 검토를 수행할 때의 공통 작업 원칙을 정의합니다.

## 1. 공통 협업 원칙

- 언어와 스택: Python 3.10+, 저장소의 기존 아키텍처 및 디렉토리 경계를 따릅니다.
- 설정 제약: `.env`를 통해 통합 관리합니다 (`.env.example` 참고).
- 코드 품질: 실행 가능성, 회귀 검증 가능성, 추적 가능성(로그/오류 메시지 명확성)을 최우선으로 합니다.
- 스타일 제약:
  - 줄 너비 120
  - `black` + `isort` + `flake8`
  - 핵심 변경에는 최소한 문법 검증이 필요합니다 (`py_compile`) 또는 대응 테스트 검증.
  - 새로 추가하거나 수정한 코드 주석은 반드시 영어로 작성해야 합니다.
- Git 제약:
  - 명시적 확인 없이 `git commit`을 실행하지 않습니다.
  - commit message에 `Co-Authored-By`를 추가하지 않습니다.
  - 이후 모든 commit message는 반드시 영어로 작성합니다.

## 2. Issue 분석 원칙

각 Issue는 먼저 다음 3가지 질문에 답해야 합니다:

1. 타당한가 (Reasonable)
- 실제 영향(기능 오류, 데이터 오류, 성능/안정성 문제, 사용성 저하)을 설명하고 있는가.
- 검증 가능한 근거(로그, 스크린샷, 재현 절차, 버전 정보)가 있는가.
- 프로젝트 목표(주식 분석, 데이터 소스, 알림, API/WebUI, 배포 경로)와 관련이 있는가.

2. 실제 Issue인가 (Valid Issue)
- 버그/기능 누락/회귀/문서 오류 중 하나에 해당하며, 단순 문의나 환경 오용이 아닌가.
- 저장소 책임 범위 내에서 원인을 특정할 수 있는가. 서드파티 서비스 장애라도 저장소 측 안전망이 필요한지 판단합니다.
- 사용 방법 문제라면 코드 버그가 아닌 문서 개선 또는 FAQ로 전환합니다.

3. 해결 가능성이 높은가 (Solvability)
- 안정적으로 재현 가능한가.
- 의존성을 제어할 수 있는가 (서드파티 API, 네트워크, 권한, 키).
- 변경 범위와 위험 등급 (낮음/보통/높음).
- 임시 완화 방안이 있는가 (다운그레이드, 안전망, 토글, 재시도, 롤백 전략).

### Issue 결론 템플릿

- 결론: `성립 / 부분 성립 / 불성립`
- 분류: `bug / feature / docs / question / external`
- 우선순위: `P0 / P1 / P2 / P3`
- 난이도: `easy / medium / hard`
- 권장 조치: `즉시 수정 / 일정 수정 / 문서 명확화 / 닫기`

## 3. PR 분석 원칙

각 PR은 다음 순서로 검토합니다:

1. 필요성 (Necessity)
- 명확한 문제를 해결하거나 명확한 비즈니스 가치를 제공하는가.
- "변경을 위한 변경"식 리팩토링을 피하고 있는가.

2. 추적 가능성 (Traceability)
- 대응하는 Issue와 연결되어 있는가 (권장: `Fixes #xxx` 또는 `Refs #xxx`).
- Issue가 없는 경우 PR 설명에 동기, 시나리오, 검수 기준이 명시되어 있는가.

3. 유형 판정 (Type)
- 명확히 표기: `fix / feat / refactor / docs / chore / test`.
- `fix/bug` 유형 PR: 원래 문제, 근본 원인, 수정 내용, 회귀 위험을 반드시 설명해야 합니다.

4. 설명 완전성 (Description Completeness)
- 반드시 포함해야 할 항목:
  - 배경과 문제
  - 변경 범위 (수정된 모듈)
  - 검증 방법과 결과 (명령어, 핵심 출력)
  - 호환성 및 호환성 깨짐 변경 설명 (있는 경우)
  - 롤백 방안 (최소 한 문장)
  - Issue 수정인 경우: PR 설명에 닫기 구문 명시 (예: `Fixes #241` / `Closes #241`)

5. 병합 가능 여부 (Merge Readiness)
- 즉시 병합 가능 (Ready to Merge) 조건:
  - 목표가 명확하고 필요한 변경
  - Issue 또는 동등한 수준의 문제 설명 존재
  - 변경 내용과 설명이 일치하며 숨겨진 부작용 없음
  - 핵심 검증 통과 (문법/테스트/핵심 경로)
  - 차단성 위험 없음 (보안, 데이터 손상, 명백한 성능 저하)
- 즉시 병합 불가 (Not Ready) 조건:
  - 설명이 불완전하여 동기와 영향을 확인할 수 없음
  - 검증 근거 없음
  - 명백한 위험이 있으나 롤백 전략 없음
  - 저장소 방향과 무관하거나 중복 구현

## 4. 전달 및 릴리스 동기화 원칙

- 기능 개발, 버그 수정 완료 후 반드시 문서를 동기화합니다:
  - `README.md` (사용자에게 보이는 기능, 사용 방법, 설정 항목 변경)
  - `docs/CHANGELOG.md` (버전 변경 기록, 영향 범위, 호환성 설명)
- 자동 버전 태그는 기본적으로 **트리거되지 않으며**, 커밋 설명에 대응 태그를 명시적으로 추가해야 tag가 생성됩니다:
  - `#patch`: 수정류, 소규모 변경 (+0.0.1)
  - `#minor`: 하위 호환 기능 추가 (+0.1.0)
  - `#major`: 호환성 깨짐 변경 또는 대규모 아키텍처 조정 (+1.0.0)
  - 태그를 추가하지 않으면 기본적으로 tag를 만들지 않음
- 기존 Issue를 해결하는 변경이라면, commit 또는 PR 설명에 닫기 구문을 반드시 선언해야 합니다 (`Fixes #xxx` / `Closes #xxx`). 수정 완료 후 Issue가 미결 상태로 남지 않도록 합니다.

### Tag 및 Release 규칙

**Tag는 반드시 annotated tag(`-m` 주석 포함)를 사용해야 하며, 경량(lightweight) tag는 금지합니다.**

```bash
# 올바름: annotated tag, -m 내용이 GitHub Release 본문이 됩니다
git tag -a v3.x.x -m "Bug fixes:
- fix(xxx): 설명 (#issue)

Features:
- feat(xxx): 설명 (#issue)"
git push origin v3.x.x
```

- CI(`docker-publish.yml`)는 tag에 비어 있지 않은 주석이 있는지 검증하며, 경량 tag는 바로 실패합니다.
- `.github/workflows/create-release.yml`은 tag 푸시 후 **자동으로 주석 내용으로 GitHub Release를 생성**합니다. 수동 편집이 필요 없습니다.

**GitHub Release의 "What's Changed" 자동 생성 규칙(`.github/release.yml`):**

- 내용 출처: 두 tag 사이에 **병합된 PR**, PR label 기준으로 분류됩니다.
- 직접 `git commit`으로 푸시된 커밋은 **자동 생성 내용에 포함되지 않습니다**.
- 따라서 팀 규칙:
  - 사용자에게 보이는 기능/수정은 **PR**을 통해 병합하고 label을 명확히 붙입니다 (`bug` / `enhancement` / `data-source` 등).
  - 직접 커밋을 푸시한 경우(긴급 수정 등), tag의 `-m` 주석에 **해당 변경 사항을 수동으로 보충**하여 Release Notes가 완전하도록 합니다.

**`docs/CHANGELOG.md` 유지보수 규칙:**

- 새 버전 항목 추가 시, `[Unreleased]` 아래 내용을 `## [X.Y.Z] - YYYY-MM-DD` 로 이동하고 `[Unreleased]`를 초기화합니다.
- tag 생성 전 CHANGELOG 동기화를 강제하지는 않지만 (CI가 tag 주석 검증으로 변경됨), 동기화 유지를 권장합니다.

## 5. 리뷰 출력 형식

### Issue 리뷰 출력

- `타당한가`: 예/아니오 + 이유
- `실제 Issue인가`: 예/아니오 + 이유
- `해결 가능성이 높은가`: 예/아니오 + 난점
- `권장 조치`: 수정 / 일정 수정 / 문서 / 닫기

### PR 리뷰 출력

- `필요성`: 통과 / 미통과
- `대응 Issue 여부`: 있음 / 없음 (번호)
- `PR 유형`: fix / feat / ...
- `description 완전성`: 완전 / 불완전 (누락 항목)
- `즉시 병합 가능 여부`: 가능 / 불가 + 필수 수정 항목
